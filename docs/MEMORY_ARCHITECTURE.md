# Project Brain — Memory Architecture

How the six pillars — **Obsidian**, **Markdown**, **Embeddings**, **Vector DB**,
**Knowledge Graph**, and **Claude Code** — interlock to form a durable, local
memory layer.

The unifying principle: **Markdown is canonical; everything else is a derived,
rebuildable index.** Lose any derived store and you lose nothing but search speed
until the next rebuild. Lose the Markdown and you've lost the actual memory.

---

## 1. The six pillars and their roles

| Pillar | Role | Mutable by human? |
|---|---|---|
| **Obsidian** | The editor and UX layer. Brain never depends on Obsidian running; it only respects Obsidian's conventions (folders, frontmatter, `[[wikilinks]]`, tags). | Yes — it's the editor. |
| **Markdown** | Canonical storage format for every memory. Plain text, diffable, future-proof. | Yes — the source of truth. |
| **Embeddings** | Meaning-based retrieval. Turns text into vectors so "auth" finds notes about "authentication" even without the exact word. | No — derived. |
| **Vector DB** | Stores embeddings + metadata; answers "most similar to X". | No — derived. |
| **Knowledge Graph** | Stores note-link/tag/entity relationships; answers "what's connected to X". | No — derived. |
| **Claude Code** | The consumer and producer. Reads memory on demand, writes memory back. Via MCP. | N/A — the client. |

## 2. The note model

Every memory is a Markdown file with this shape:

```markdown
---
id: 01HXYZ...                # ULID — stable, sortable, URL-safe
title: Why we chose LanceDB
type: decision               # decision | concept | bug | howto | log | note
tags: [storage, vector-db]
source: claude-code          # who/what created it
created: 2026-07-17T09:00:00Z
updated: 2026-07-17T09:00:00Z
links: [architecture/storage]  # explicit, in addition to [[wikilinks]]
---

# Why we chose LanceDB

LanceDB is local-first, file-based, and needs no server process —
matching our local-first principle. See [[architecture/tech-stack]].

Embeddings are generated by [[embeddings/nomic-embed]] via Ollama.
```

- **Frontmatter** is machine-readable metadata — the index source for tags,
  type, dates, and explicit relations.
- **Body** is human-readable prose — the index source for embeddings and FTS.
- **`[[wikilinks]]`** are Obsidian-native and double as graph edges.

## 3. How they work together — the lifecycle

### 3.1 Ingestion (vault → indexes)
```
Markdown note
   │
   ├──▶ Storage.parse()      →  Note { id, frontmatter, body, links, tags }
   │
   ├──▶ Chunker.split()      →  Chunk[]  (heading-aware, overlap)
   │         │
   │         └──▶ Embedder.embed()  →  Vector[]
   │                   └──▶ VectorStore.upsert()
   │
   └──▶ GraphBuilder.build() →  Nodes + Edges
                    │
                    ├── links ────────────▶ GraphStore.upsertEdges()
                    ├── tags ─────────────▶ GraphStore.upsertTags()
                    └── frontmatter.relations ▶ GraphStore.upsertEdges()
```
The Indexer records `hash(note) → indexed` so unchanged notes are skipped on
re-index. This is what makes incremental indexing cheap.

### 3.2 Retrieval (query → context)
```
Claude's question
   │
   ├──▶ Search.semantic(query)   ── embed query, kNN ──▶ ranked chunks
   ├──▶ Search.lexical(query)    ── FTS5 match ─────────▶ ranked chunks
   │            (merged + re-ranked by Search.hybrid)
   │
   ├──▶ Graph.expand(top hits)   ── neighbor traversal ─▶ related notes
   │
   └──▶ Context.build(intent)    ── budget + dedupe ───▶ context bundle
                                                 + manifest (citations)
```

### 3.3 Production (Claude → vault)
```
Claude learns something worth keeping
   │
   └──▶ Remember.write({ title, content, tags, links })
            │
            └──▶ Storage writes Markdown atomically
                     │
                     └──▶ File Watcher fires
                            │
                            └──▶ Indexer re-processes that one note
                                   (chunks → vectors → graph), async
```

The key property: **a write is durable the moment the Markdown lands**, because
Markdown is canonical. Searchability follows milliseconds later.

## 4. The three retrieval modalities, and why we need all three

| Modality | Catches | Misses |
|---|---|---|
| **Semantic** (vectors) | Paraphrases, concepts, "auth"↔"authentication" | Exact identifiers, rare terms, code symbols |
| **Lexical** (FTS) | Exact strings, error messages, file names, code | Paraphrases, synonyms |
| **Graph** (traversal) | Relationships, "what else is connected to this" | Topical relevance on its own |

`brain_search_hybrid` runs semantic + lexical and merges; `brain_build_context`
adds graph expansion on top. No single modality is enough — that's why Brain
maintains all three indexes.

## 5. Chunking strategy

- **Heading-aware, not fixed-window.** A note is split along Markdown headings
  first; oversized sections are sub-chunked with overlap. This keeps each chunk
  coherent (a chunk about one thing) which improves both embedding quality and
  citation.
- **Chunk carries provenance:** `{ noteId, headingPath, charStart, charEnd }`.
- **Defaults are configurable** (target size, overlap) and tuned in Phase 2.

## 6. Embeddings

- **Model:** `nomic-embed-text` via **Ollama** by default — fully local, good
  quality/size tradeoff, and `IEmbedder` makes it swappable.
- **Dimensionality** is recorded per-store so a model swap is detected and forces
  a rebuild (vectors of different dimensions can't coexist).
- **Batching:** the Embedder batches chunks to bound Ollama round-trips.
- **Deterministic fakes** in tests — no network, no model needed for unit tests.

## 7. Vector store (LanceDB)

- **Schema:** `table memories { id, note_id, chunk_text, embedding, heading_path, tags, mtime }`.
- **Index:** ANN index for fast kNN.
- **Filter pushdown:** tag/date filters applied in the query, not after, to keep
  result sets small.
- **Rebuildable:** dropping the table and re-indexing reproduces it exactly.

## 8. Knowledge graph (SQLite)

- **Nodes:** one per note `{ id, note_id, title, type }`.
- **Edges:** `{ src, dst, relation }` where relation ∈
  `{ wikilink, tag, frontmatter }`.
- **Tags** as first-class (a tag links all notes sharing it).
- **Queries:** neighbors, paths, connected components, community detection.
- **Co-located with FTS** in one SQLite DB (decision tracked in
  `memory/decisions.md`) for simplicity — both are "structured metadata about
  notes".

## 9. Claude Code integration

- Claude Code is an **MCP client**. Brain is an **MCP server**.
- **On-demand, not preload:** Claude calls `brain_build_context` or
  `brain_search_hybrid` when it needs memory — it does not ingest the whole vault
  into context.
- **Write-back is explicit:** Claude decides what's worth remembering and calls
  `brain_remember`. Brain never auto-writes without an extractor proposing and a
  confirmation path (see Group F tools).
- **Provenance in, provenance out:** results carry source paths so Claude can
  cite memory in its reasoning and the user can verify.

## 10. Privacy & local-first guarantees

- Embeddings run via local Ollama. **No text leaves the machine for indexing.**
- Brain makes **no outbound network calls** unless the user opts into a remote
  capability (e.g., a remote embedder) in config.
- The vault, vector DB, and graph DB all live under a single user-chosen data
  directory. Nothing is uploaded.
- No telemetry, no analytics, no "phone home". Verifiable by audit: the only
  network-capable code is behind `IEmbedder` and an off-by-default remote flag.

## 11. Consistency model

- **Vault is the single source of truth.** If the vault and a derived store
  disagree, the vault wins; a rebuild reconciles.
- **Eventual consistency between write and search:** a just-written note is
  recallable by ID immediately, and by search once the async indexer processes
  it (bounded, debounced).
- **Crash safety:** Storage writes atomically (temp file + rename). The indexer
  is resumable — a crash mid-index leaves a `pending` set that resumes on next
  start, never a half-indexed note in the vector store (per-note upserts are
  transactional).

## 12. Failure modes & recovery

| Failure | Recovery |
|---|---|
| Vector DB corrupted | `brain_rebuild --full` from vault. |
| Graph DB corrupted | Same — rebuild from vault. |
| Embedding model changed | Detected via dimensionality mismatch → forces rebuild. |
| Vault file deleted | Gone from indexes on next index pass; `brain_forget` had archived if used. |
| Ollama down at index time | Indexer queues + retries; search falls back to lexical-only with a warning. |
| Partial write crash | Atomic write means either the old or new file exists, never a torn one. |

## 13. Why this architecture is durable

- **Format longevity:** Markdown outlives any specific tool. The memory survives
  Brain itself.
- **No vendor lock-in:** every derived store has a pluggable interface.
- **Reproducibility:** anyone can rebuild identical indexes from the same vault
  + same model.
- **Auditability:** a human can read every memory; nothing is hidden in a binary
  blob.
- **Extensibility:** new retrieval modalities (e.g., a code-symbol index) become
  a new derived store behind a new port — no rewrite of the others.
